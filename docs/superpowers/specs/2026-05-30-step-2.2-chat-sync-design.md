## Goal

Two real Blab accounts on two devices can send and receive messages live (~2s), survive app restart, and read receipts reflect actual partner reading.

## Product decisions locked in this brainstorm

| # | Decision |
|---|---|
| 1 | **Read receipt trigger:** scroll-into-view. Server marks per-message read when the partner's client reports the message id has been on-screen (not just when chat opened). Matches Signal stance. |
| 2 | **Mock chats wiped.** `kMockChats` / `kAswinMockChats` removed from the build. New accounts land on real empty state + "Invite someone" CTA. |
| 3 | **Chat tile appears at invite claim**, before any message. Both parties see the partner's tile + empty-state exchange card the moment Aswin signs up via the invite. |
| 4 | **Profile photos deferred** to a dedicated sub-step right after chat sync ships. Until then, gradient-initial avatars everywhere. |
| 5 | **Encryption deferred** to Step 2.6. Plaintext on the server until then. **Hard gate:** no one outside us two gets the app until 2.6 is in. Plaintext history from dev may be wiped or left unreadable at the switch — accepted. |

## Data model (Postgres, RLS on every table)

- `profiles` — one row per `auth.users` id. Display name, avatar bucket key (null for now), interface_language, created_at. Auto-created via trigger on `auth.users` insert.
- `chats` — id (uuid), created_at. No "owner" — symmetric.
- `chat_members` — `chat_id` × `user_id`. Composite PK. `learning_language` (the language *this* member is learning from the *other* member — per-chat per-user, see Resolved Q1). `joined_at`.
- `messages` — id (uuid), chat_id, sender_id, body (text, plaintext for now), created_at (server time, source-of-truth ordering), edited_at (nullable), reply_to (nullable fk to messages.id), deleted_at (soft delete for Undo / Delete-forever).
- `message_reads` — `message_id` × `user_id` (the reader). `read_at`. Composite PK. Insert-only.

RLS rules (sketch — final policies in migration):
- `chats`: SELECT only if `auth.uid()` is in `chat_members` for that chat.
- `messages`: SELECT/INSERT only if sender's chat membership includes `auth.uid()`. UPDATE/DELETE only by sender.
- `message_reads`: INSERT only by the reader (`auth.uid() = user_id`), and only for messages in chats they are a member of.

## Real-time

Supabase Realtime channels:
- `chat:<chat_id>` — INSERT/UPDATE/DELETE on `messages` and INSERT on `message_reads` filtered to this chat. Each client subscribes only to chats they're a member of.
- `chat-list:<user_id>` — listens for new `chat_members` rows for this user (so Nastia's list updates when Aswin claims an invite).

## Flutter side — chunks

### Chunk A: schema + auth wiring (no visible change)
- Supabase migration adds tables, RLS, triggers, publication.
- `messages` + `chats` added to `supabase_realtime` publication.
- New `lib/shared/services/chat_service.dart` exposes typed CRUD + stream.
- Mock chat data deleted.
- Empty chat list state shown for both real accounts.

### Chunk B: send + receive live
- `ChatNotifier` rewired: `addOutgoing` writes to Postgres; UI optimistically inserts a "pending" bubble, flips to "delivered" when the row insert returns, then "read" when a `message_reads` row arrives.
- Subscription delivers partner messages within 2s.
- `VisibilityDetector` (or scroll listener) reports message ids that fully enter the viewport → batched insert into `message_reads`.
- Chat list tile shows unread = badge + **bold name** (locked earlier: WhatsApp pattern).
- Per-chat learning language stored in `chat_members.learning_language` and surfaced via the existing learning-language sheet (US-022).

### Chunk C: restart + offline
- Cold launch hydrates chat list + last 50 messages per opened chat from Postgres.
- Offline-detected sends → local "pending" state; on reconnect, the queue flushes to Postgres in order.
- Failed inserts (5xx) show ⚠ + retry sheet (already built in Step 1.10, just rewire).

## Risk register

- **Realtime publication forgotten in dashboard.** Verified by checking `supabase_realtime` publication includes `messages` and `chats` before declaring Chunk A done.
- **RLS too strict — partner can't see message.** Caught by Chunk B test step 4.
- **RLS too loose — third account can read.** Will spin up a temp throwaway account during Chunk B and confirm it sees nothing.
- **Scroll-into-view double-counts reads.** `message_reads` composite PK on (`message_id`, `user_id`) makes the insert idempotent — second insert is a no-op.
